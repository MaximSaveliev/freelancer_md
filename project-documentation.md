# Полная техническая документация проекта Freelancer.md

**Версия:** 1.1.0
**Дата:** 26 Марта 2026
**Статус:** Активная разработка / Production Ready

---

## 1. Обзор проекта
**Freelancer.md** — это современная биржа фриланса, объединяющая заказчиков и исполнителей. Платформа поддерживает двойные роли (один аккаунт для заказчика и фрилансера), систему тендеров (аукционы со скрытыми ставками), безопасные сделки (Escrow) и систему премиальных подписок для исполнителей.

---

## 2. Стек технологий

### 2.1. Frontend (Клиентская часть)
* **Фреймворк:** Next.js (App Router), React.
* **Стилизация:** Tailwind CSS.
* **Анимации:** Framer Motion (motion/react).
* **Иконки:** Lucide React.
* **Особенности:** Адаптивный дизайн, темная тема (Dark Mode), интерактивные дашборды, модальные окна.

### 2.2. Backend (Серверная часть)
Бэкенд построен на гибридной/микросервисной архитектуре для обеспечения максимальной производительности и удобства интеграций:
* **Core API:** **.NET** (C#). Отвечает за основную бизнес-логику, маршрутизацию и работу с данными.
* **Real-time:** **SignalR**. Используется для мгновенных уведомлений, обновления статусов аукционов и чатов в реальном времени.
* **Payment & Webhook Service:** **FastAPI** (Python). Выделенный микросервис для обработки платежей и вебхуков.
* **База данных:** **PostgreSQL** (основное хранилище реляционных данных).
* **BaaS / Storage:** **Supabase**. Используется для хранения файлов (портфолио, аватары) и, опционально, аутентификации.

### 2.3. Инфраструктура и DevOps
* **Хостинг:** **LiquidWeb** (выбран за прозрачные тарифы и оптимальное соотношение цена/качество).
* **Контейнеризация:** **Docker**. Весь проект (фронтенд, бэкенд-сервисы, БД) упакован в Docker-образы.
* **Оркестрация:** **Docker Compose** для поднятия и связывания контейнеров на сервере.
* **CI/CD:** **GitHub Actions**. Настроены пайплайны для автоматической сборки, тестирования и деплоя на сервер.
* **Сеть и Безопасность:** **Cloudflare**. Обеспечивает SSL-сертификаты (HTTPS), защиту от DDoS, WAF (Web Application Firewall), кэширование и сбор метрик (анализ трафика, статусов ответов, блокировок).

### 2.4. Сторонние интеграции
* **Платежный шлюз:** **Stripe**. Обработка подписок и холдирование средств (Escrow).
* **Email-провайдер:** **Resend**. Транзакционные письма (OTP, уведомления о победе в тендере, системные алерты).

## 3. Архитектура Бэкенда

Основное API на **.NET** построено по классической **Слоистой архитектуре (Layered Architecture)**:
1. **PL (Presentation Layer):** RESTful API контроллеры. Принимают HTTP-запросы, валидируют входные данные (DTO) и возвращают HTTP-ответы.
2. **BLL (Business Logic Layer):** Сервисный слой. Содержит всю бизнес-логику (правила аукционов, расчет комиссий, логика подписок).
3. **DAL (Data Access Layer):** Слой доступа к данным. Работает с PostgreSQL через Entity Framework Core, выполняет CRUD операции.

### 3.1. Событийно-ориентированная обработка платежей (Event-Driven Webhooks)
Интеграция со Stripe реализована через асинхронную обработку событий:
1. Пользователь инициирует оплату на клиенте -> Запрос идет на API.
2. Stripe обрабатывает платеж и шлет асинхронный **Webhook**.
3. **FastAPI** сервис принимает вебхук от Stripe, валидирует подпись (Signature).
4. FastAPI триггерит внутренний **Event** (событие) в системе (например, через Message Broker или прямой вызов внутреннего API .NET).
5. Система обновляет статус подписки пользователя или статус заморозки средств (Escrow) в БД.

## 4. Ключевая Бизнес-Логика

### 4.1. Система Аукционов (Tenders & Sealed Bids)
* **Создание:** Заказчик публикует проект со статусом "Аукцион" и указывает вилку бюджета.
* **Ставки (Bids):** Фрилансеры предлагают свою цену, срок и сопроводительное письмо.
* **Приватность:** Ставки скрыты (Sealed). Фрилансер видит только количество заявок. Пользователи с тарифом Premium видят *среднюю* предложенную цену.
* **Выбор исполнителя:** Заказчик видит все ставки в удобном дашборде. При выборе исполнителя срабатывает триггер заморозки средств.

### 4.2. Безопасная сделка (Escrow)
1. Клиент выбирает фрилансера.
2. Stripe холдирует (замораживает) сумму на карте клиента.
3. Проект переходит в статус `IN_PROGRESS`.
4. После завершения работы и апрува клиентом, бэкенд дает команду Stripe на списание средств (Capture) и зачисление их на баланс фрилансера (за вычетом комиссии платформы).

### 4.3. Система Подписок (Subscriptions)
Фрилансеры могут улучшить свой аккаунт:
* **Free:** Базовый функционал, 2 отклика в неделю, до 5 навыков.
* **PRO ($9.99/мес):** Бейдж PRO, приоритет в поиске, 5 откликов, до 10 навыков.
* **Premium ($19.99/мес):** Эксклюзивный бейдж, аналитика ставок (Bid Insights), закрепление в топе поиска.
* *Техническая реализация:* Подписки управляются через Stripe Billing. Вебхуки `invoice.paid` и `customer.subscription.deleted` автоматически обновляют права пользователя в PostgreSQL.

### 4.4. Мультиролевая система
* Пользователь имеет единый аккаунт (один Email/Password).
* В БД хранятся два связанных профиля: `ClientProfile` и `FreelancerProfile`.
* Переключение ролей происходит мгновенно на клиенте, API запросы авторизуются на основе текущего активного контекста (передается в заголовках или JWT токене).

## 5. Структура Базы Данных (PostgreSQL)

* **Users:** `id`, `email`, `password_hash`, `created_at`.
* **Profiles:** `id`, `user_id`, `role_type` (Client/Freelancer), `name`, `avatar_url`, `rating`.
* **Freelancer_Details:** `profile_id`, `skills` (JSONB), `bio`, `rate`, `subscription_tier`.
* **Projects:** `id`, `client_id`, `title`, `description`, `budget_min`, `budget_max`, `status`, `is_auction`.
* **Bids:** `id`, `project_id`, `freelancer_id`, `price`, `timeframe`, `cover_letter`, `status` (Pending/Accepted/Rejected).
* **Transactions:** `id`, `user_id`, `stripe_payment_intent_id`, `amount`, `status`, `type` (Escrow/Subscription/Withdrawal).

## 6. Мониторинг и Безопасность

1. **Cloudflare:**
   * Проксирование всего трафика.
   * WAF (Web Application Firewall) блокирует подозрительные SQLi и XSS атаки.
   * Метрики: дашборды Cloudflare используются для анализа географии запросов, процента успешных ответов (200 OK) и ошибок (4xx, 5xx).
2. **SSL/TLS:** Строгое HTTPS соединение от клиента до Cloudflare и от Cloudflare до LiquidWeb серверов.
3. **Docker Network:** База данных PostgreSQL и внутренние микросервисы не торчат наружу, общение происходит внутри закрытой Docker-сети. Наружу открыты только порты API (через Nginx/Traefik reverse proxy).
4. **SignalR Security:** Подключение к вебсокетам требует валидного токена авторизации.
